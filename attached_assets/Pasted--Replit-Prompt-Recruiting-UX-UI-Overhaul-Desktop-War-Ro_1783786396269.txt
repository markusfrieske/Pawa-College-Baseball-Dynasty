# Replit Prompt: Recruiting UX/UI Overhaul - Desktop War Room + Mobile Parity

Use this as a focused design and implementation prompt for the recruiting experience in `markusfrieske/Pawa-College-Baseball-Dynasty`.

## Product Direction

This game should be optimized first for desktop because the long-term goal is a PC/Steam release. Desktop should feel like a serious college baseball recruiting command center: dense, fast, analytical, keyboard/mouse friendly, and built for repeated weekly use.

Mobile should not be a stripped-down or separate recruiting mode. Mobile must contain the same decision-critical information, filters, actions, and recruiting intelligence as desktop, reorganized into a phone-friendly layout.

Current issue: mobile recruiting is too condensed and removes too much context, making it difficult or impossible to recruit properly. Also, desktop needs a stronger War Room/dashboard experience that helps coaches make smart decisions instead of forcing them to manually scan everything.

## Core Goal

Reimagine recruiting as a unified Recruiting Hub with:

- a rich desktop War Room dashboard
- a desktop recruiting board/table for power users
- a mobile recruiting board with full feature parity
- shared data and components so mobile and desktop cannot drift apart
- clear weekly decision support: who to scout, who to call, who to offer, who is slipping, who is affordable, who fits roster needs

## Non-Negotiables

- Desktop remains the premium PC experience.
- Mobile must allow a coach to complete a full recruiting week without switching to desktop.
- Mobile must include every core desktop recruiting action.
- Mobile must include every decision-critical desktop field.
- Do not make a separate reduced mobile recruiting system.
- Do not hide essential information only because the screen is narrow.
- Do not use tiny unreadable text to force desktop density onto mobile.
- Prefer responsive reflow, expandable sections, sticky bars, drawers, and bottom sheets.

## UX Principle: Progressive Disclosure, Not Information Removal

Desktop can show more information at once. Mobile can show information in layers.

That means:

- desktop uses tables, side panels, pinned columns, hover details, and multi-panel dashboards
- mobile uses expandable cards, sticky controls, tabs, bottom sheets, accordions, and action trays
- both show the same underlying information and support the same actions

Mobile should feel like the same game, not a companion app.

## Desktop Recruiting Hub / War Room

Create or redesign the desktop War Room as the main weekly recruiting dashboard.

### Layout

Use a desktop-first dashboard layout:

- top command header
- left recruiting board / priority list
- right contextual detail panel
- bottom or side modules for intelligence and weekly planning
- dense but readable cards, not oversized marketing panels

### War Room Modules

Build the War Room as practical decision support, not decoration.

#### 1. Weekly Command Header

Show:

- current phase and week
- recruiting points used/remaining
- scout points used/remaining
- visits used this season
- head coach visits used
- NIL budget remaining
- commits count
- projected roster size
- target count
- ready status

This header should be sticky on desktop and mobile.

#### 2. Recommended Actions

A ranked list of the highest-value next actions.

Each recommendation should include:

- recruit name
- position
- star rating
- current interest
- action: scout, email, phone, visit, HCV, offer, hold
- expected reason
- urgency
- cost
- risk if ignored

Examples:

- "Offer now: rival offer + Top 3 stage"
- "Phone call: high fit, trailing by 8"
- "Scout: high upside but low certainty"
- "Visit: 5-star pitcher, late-stage battle"
- "Hold: already contacted this week"

#### 3. Hot Battles

Show contested recruits where multiple teams are active.

Fields:

- recruit
- user interest
- leader
- margin
- top 3 schools
- offers out
- recent rival actions
- current stage
- recommended counter move

#### 4. At-Risk Targets

Show recruits where the coach is losing ground.

Fields:

- recruit
- current rank
- leader
- gap
- recent trend
- stage
- has offer?
- recommended action

#### 5. Best Fits

Surface undervalued or high-fit recruits.

Factors:

- roster need
- recruit quality
- proximity
- school strengths
- coach archetype
- pitch priority match
- NIL affordability
- low rival pressure

#### 6. Roster Needs

Show next-year roster needs by position group.

Include:

- current players
- seniors/departures
- commits
- projected depth
- recommended target count
- shortage warnings

Position groups:

- pitchers
- catchers
- infield
- outfield
- utility/depth

#### 7. NIL Watch

Show budget pressure.

Include:

- NIL remaining
- committed NIL
- top targets by NIL cost
- unaffordable recruits
- value targets
- warning when offers would exceed budget

#### 8. Visit Planner

Show:

- visits used / cap
- recruits already visited
- best visit candidates
- head coach visit candidates
- visit cost by recruit
- recommended week to use visit

#### 9. Offer Board

Show:

- recruits with offer
- recruits ready for offer
- recruits who should not be offered yet
- no-offer leaders who could steal attention but should not be signable
- offer urgency

#### 10. Recent Activity

Show a timeline:

- user's actions
- rival actions, where public/allowed
- stage changes
- decommits
- commitments
- recruiting news

- type

- stage

- scout %

- NIL

### Mobile Actions

Every desktop action must be available on mobile:

- target / untarget
- set board rank
- scout
- email
- phone
- campus visit

- offer scholarship

Every disabled action must explain why:

- not enough points

- wrong phase

## Shared Data / Component Architecture

Avoid two separate recruiting products.

Implement a shared data layer:

- `useRecruiting()` returns a complete board model
- desktop and mobile consume the same normalized recruit view model
- recommendation logic is shared
- action availability logic is shared
- disabled reason logic is shared
- filters/sorts are shared

Suggested shared model:

```ts
type RecruitingBoardRecruit = {
  id: string;
  identity: RecruitIdentity;
  rankings: RecruitRankings;
  scouting: ScoutingSummary;
  ratings: RatingsSummary;
  interest: InterestSummary;
  battle: BattleSummary;
  nil: NilSummary;
  rosterFit: RosterFitSummary;
  actions: ActionAvailability;
  recommendation: RecommendedAction;
};
```

This prevents desktop showing data that mobile cannot access.

## Reimagined Recruiting Dashboard Information Hierarchy

Priority order for what coaches need to know:

1. What do I need this week?
2. Who am I trying to sign?
3. Who am I about to lose?
4. Who can I afford?
5. Who fits my roster?
6. Who needs scouting before I spend points?
7. Which action gives the best return?
8. What did rivals do?
9. Am I ready to advance?

The dashboard should answer those questions without making the user open 20 recruit profiles.

## Acceptance Criteria

### Desktop

- War Room exists and is useful on desktop.
- A coach can decide a complete recruiting plan from the War Room.
- Desktop recruiting board still supports power-user scanning and sorting.
- No information currently needed for recruiting is removed.
- The layout works at 1440 and 1920 px widths.

### Mobile

- A coach can complete a full recruiting week from mobile.
- Mobile exposes the same actions as desktop.
- Mobile exposes the same decision-critical fields as desktop.
- Mobile filters and sorting match desktop.
- Mobile recruit detail has full analytical depth.
- Text does not overlap or clip.
- Controls are tappable.
- Disabled actions explain why.
- Mobile works at 360, 390, 414, and 430 px widths.

### Cross-Platform

- Desktop and mobile use the same APIs.
- Desktop and mobile use the same action availability logic.
- Desktop and mobile generate the same logs/state changes.
- Mobile should not get stale when desktop fields change.

## Manual QA Checklist

Run through this as a human coach:

1. Open desktop recruiting.
2. Use War Room to identify top recommended actions.
3. Filter to pitchers.
4. Target three recruits.
5. Set board ranks.
6. Scout one recruit.
7. Email one recruit.
8. Phone one recruit.
9. Schedule a visit.
10. Offer a scholarship.
11. Open recruit detail and verify ratings, priorities, NIL, top schools, and action history.
12. Switch to mobile width.
13. Repeat every step without using desktop.
14. Confirm the same data and action results appear on both views.
