Problems To Fix
1. Final Signing Day Can Award Recruits Without Offers
Current issue:
The final signing-day pass can fall back from "highest offered team" to "highest raw interest team." That means a coach can win a recruit without making an offer.
Required behavior:
A recruit should only sign with a team that has a valid offer unless the route is an explicit commissioner-only override.
The winning team must meet the recruit's signing threshold.
NIL budget and roster room must be checked.
Tiebreakers must be deterministic.
Implementation direction:
Create one canonical server-side resolver, for example:
type RecruitingResolution = {
  recruitId: string;
  winnerTeamId: string | null;
  winnerScore: number;
  runnerUpTeamId: string | null;
  runnerUpScore: number;
  wonBy: number | null;
  eligibleTeams: Array<{
    teamId: string;
    teamName: string;
    score: number;
    hasOffer: boolean;
    metThreshold: boolean;
    nilAffordable: boolean;
    rosterAllowed: boolean;
    rank: number;
  }>;
  reason: string;
};
Use this resolver in:weekly recruiting advancement
signing-day preview
final signing-day finalization
any manual signing endpoint if retained

Suggested function name:
resolveRecruitSigningWinner(...)
Resolver rules:
Only consider recruiting_interests rows for teams in the same league.
Only consider teams with hasOffer === true.
Only consider teams that meet the recruit's current signing threshold.
Only consider teams that can afford NIL.
Only consider teams that have roster room.
Sort by:final interest score descending
hasOffer first, though all eligible should have offer
higher board rank priority if present
better recruit/team fit score if available
earlier offer/action timestamp if available
stable team ID as final deterministic tiebreak

Do not use random selection for final signing.
2. Manual /sign Endpoint Is Too Permissive
Current issue:
The manual recruit sign endpoint checks only basic roster/NIL/already-signed conditions.
Required behavior:
Choose one:
Remove the endpoint from normal user flow.
Or make it commissioner-only.
Or make it call the canonical winner resolver and only allow the resolved winner to sign.
Minimum required checks:
Authenticated user is a coach in the league.
Recruit belongs to the league.
Team belongs to the league.
Current phase allows signing.
Team has offered the recruit.
Team meets threshold.
Team is the resolved winner.
Recruit is not already signed.
3. Recruiting Routes Need Strict League and Team Guards
Current issue:
Several routes load a recruit by ID but do not verify that the recruit belongs to the league in the URL.
Required behavior:
Every recruiting mutation must use a shared guard.
Create a helper similar to:
async function getRecruitingActionContext(req, res, options) {
  // loads league
  // loads coach for req.session.userId
  // loads team from coach.teamId
  // loads recruit
  // verifies coach.leagueId === league.id
  // verifies team.leagueId === league.id
  // verifies recruit.leagueId === league.id
  // verifies current phase if mutation
  // returns { league, coach, team, recruit }
}
Use it in:
scout
bulk-scout
phone
email
visit
head-coach-visit
offer
target
notes
board-rank
sign, if retained
recruit detail routes where user-specific interest is shown
Remove fallback logic like:
leagueTeams.find(t => !t.isCpu)
For a 14-human league, this is dangerous. If the user has no team, return 403 or 400.
4. Make Recruiting Writes Atomic and Idempotent
Current issue:
Recruiting interest rows and action counters are updated using read-modify-write patterns. This can break when 14 users or multiple tabs submit actions at the same time.
Required behavior:
Add unique constraints:
CREATE UNIQUE INDEX IF NOT EXISTS recruiting_interests_recruit_team_unique
ON recruiting_interests (recruit_id, team_id);

CREATE UNIQUE INDEX IF NOT EXISTS recruit_top_schools_recruit_team_unique
ON recruit_top_schools (recruit_id, team_id);
Update Drizzle schema to use uniqueIndex, not only index.
Add migration file.
Replace read-then-insert interest creation with upsert.
Use SQL atomic increments for:coaches.scoutActionsUsed
coaches.recruitActionsUsed
recruiting_interests.interestLevel
recruit_top_schools.accumulatedInterest

Bulk scout already has the correct idea with an atomic SQL increment. Reuse that pattern.
For weekly duplicate limits:
Phone: max 1 per recruit per team per week.
Email: max 1 per recruit per team per week.
Visit: max 1 per recruit per team per season.
Head coach visit: max 1 per recruit per team per season.
Offer: max 1 per recruit per team.
Use database-backed idempotency where practical. At minimum, re-check after insert failure and return a clean 409/400 instead of double-counting.
5. Enforce Recruiting Phase on Mutations
Recruiting actions should only work during active recruiting phases unless explicitly allowed.
Allowed active recruiting phases:
[
  "offseason_recruiting_1",
  "offseason_recruiting_2",
  "offseason_recruiting_3",
  "offseason_recruiting_4"
]
Signing reveal completion should only work after signing day results exist.
Manual signing, if retained, should only work during:
["offseason_signing_day"]
or be commissioner-only/admin-only.
6. Signing Day Reveal Needs Competitive Analytics
Current issue:
The reveal API returns player ratings and team data but not the recruiting battle result.
Add server-computed fields to each revealed recruit:
recruitingResult: {
  winnerTeamId: string;
  winnerTeamName: string;
  winnerScore: number;
  runnerUpTeamId: string | null;
  runnerUpTeamName: string | null;
  runnerUpScore: number | null;
  wonBy: number | null;
  finalRank: number;
  finalTopSchools: Array<{
    rank: number;
    teamId: string;
    teamName: string;
    score: number;
    hasOffer: boolean;
    offered: boolean;
  }>;
  actionBreakdown: {
    scout: number;
    email: number;
    phone: number;
    visit: number;
    headCoachVisit: number;
    offer: number;
    totalActions: number;
    totalInterestGained: number;
  };
  timeline: Array<{
    week: number;
    actionType: string;
    interestChange: number;
    notes: string | null;
  }>;
  offerWeek: number | null;
  campusVisitUsed: boolean;
  headCoachVisitUsed: boolean;
  nilCost: number;
  fitSummary: string;
}
How to compute:
Use recruiting_interests for final scores and offers.
Use recruiting_actions_log for action breakdown and timeline.
Use recruit_top_schools as supporting display data, not the sole source of truth.
wonBy = winnerScore - runnerUpScore.
If no runner-up, wonBy = null and display "Uncontested".
The reveal should make it obvious:
Who won the player.
How many points they won by.
Who finished second.
Whether an offer mattered.
Which actions created the gap.
Whether the player was a clean win, close battle, late flip, or uncontested signing.

Acceptance Criteria
The work is complete only when:
No recruiting route falls back to the first non-CPU team.
A user without a team cannot see or mutate another team's recruiting state.
A recruit from another league cannot be mutated through this league's URL.
Duplicate concurrent actions cannot double-spend points.
Recruiting interest rows are unique per recruit/team.
Top-school rows are unique per recruit/team.
Final signing day uses the same winner rules as preview and weekly advancement.
Recruits cannot be won without a valid offer unless using an explicit commissioner-only override.
Reveal API includes winner score, runner-up score, won-by margin, final top schools, action breakdown, and timeline.
Flipped reveal card or detail panel shows all ratings and the recruiting battle analytics.
