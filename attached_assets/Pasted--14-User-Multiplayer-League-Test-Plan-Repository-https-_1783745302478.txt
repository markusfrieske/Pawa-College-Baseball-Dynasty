# 14-User Multiplayer League Test Plan

Repository: `https://github.com/markusfrieske/Pawa-College-Baseball-Dynasty`

Purpose: run a comprehensive 14-person multiplayer test for a full college baseball dynasty league, including recruiting, regular season games, stats, standings, College World Series, playoff bracket, and championship completion.

This plan is written as a Replit runbook. Use a dedicated test database and do not run this against production data.

## 1. Test Goals

The test should prove that:

- A 14-user league can be created and completed.
- Every user owns exactly one team.
- All games simulate correctly.
- Team and player stats update correctly.
- Standings reflect real game results.
- Recruiting works when 14 users compete for the same player pool.
- Smart multi-user decisions do not corrupt state.
- The College World Series / playoff bracket generates correctly.
- Bracket advancement works through the championship.
- The final champion is recorded correctly.
- Invalid or duplicate actions are rejected safely.
- No user can affect another user's team.
- Logs and database records remain consistent from league creation through season completion.

6. Confirm you can reset or reseed the test database.

7. Confirm server logs are visible while testing.

## 7. Phase 2: Initial League State Validation

Validate the initial state before simulating anything.

Check:

- 14 teams exist.
- 14 users exist.
- Every team has one owner.
- Every team has valid roster data.
- Every player has a unique ID.
- Every team has a schedule.
- No team plays itself.
- No duplicate game IDs exist.
- Every game has valid home and away teams.
- Standings start at `0-0`.
- Player stats start at zero.
- Team stats start at zero.
- Recruiting class exists.
- Recruit IDs are unique.
- No recruit is already committed unless intended by game rules.

Suggested database checks:

```sql
SELECT COUNT(*) FROM users;
SELECT COUNT(*) FROM teams;
SELECT COUNT(*) FROM games;
SELECT COUNT(*) FROM players;
SELECT COUNT(*) FROM recruits;
```

Duplicate checks:

```sql
SELECT email, COUNT(*) FROM users GROUP BY email HAVING COUNT(*) > 1;
SELECT id, COUNT(*) FROM players GROUP BY id HAVING COUNT(*) > 1;
SELECT id, COUNT(*) FROM games GROUP BY id HAVING COUNT(*) > 1;
SELECT id, COUNT(*) FROM recruits GROUP BY id HAVING COUNT(*) > 1;
```

Ownership checks:

```sql
SELECT team_id, COUNT(*) FROM users GROUP BY team_id HAVING COUNT(*) > 1;
SELECT teams.id
FROM teams
LEFT JOIN users ON users.team_id = teams.id
WHERE users.id IS NULL;
```

Adjust table and column names to match the app schema.

## 8. Phase 3: 14-User Recruiting Simulation

This is the most important multiplayer test.

For each recruiting week:

1. Log in as Coach 01.
2. View the recruiting board.
3. Make decisions using Coach 01's strategy profile.
4. Save recruiting actions.
5. Log out.
6. Repeat for Coaches 02 through 14.
7. Advance the recruiting week.
8. Validate results.
9. Repeat until the recruiting period ends.

Recruiting validation:

- Users cannot spend more points than allowed.
- Users cannot submit negative points.
- Users cannot assign points to invalid recruits.
- Users cannot recruit for another team.
- Users cannot change another user's recruiting board.
- Recruits can receive interest from multiple teams.
- Contested recruits resolve correctly.
- Commitments go to the correct team.
- Lost recruits are marked correctly.
- Recruiting points reset or carry over according to game rules.
- Repeated submit clicks do not duplicate actions.
- Browser refresh after submit does not double-spend points.
- Actions submitted after week advancement are rejected.
- UI state matches database state.

Concurrency test:

1. Open 14 browser sessions, incognito windows, or test browser contexts.
2. Log in as one coach per session.
3. Have all coaches submit recruiting actions within 30-60 seconds.
4. Immediately advance the recruiting week.
5. Check for:
   - Missing actions
   - Duplicate actions
   - Over-budget spending
   - Incorrect commitments
   - Incorrect team IDs
   - Server errors

Pass criteria:

- All submitted recruiting actions are recorded once.
- No invalid budget states exist.
- No recruit commitment is duplicated.
- No team receives another team's recruit unless the game rules support transfers or flips.

## 9. Phase 4: Regular Season Game Simulation

Run the full regular season.

For each scheduled game day or week:

1. Confirm upcoming games are listed correctly.
2. Simulate all scheduled games.
3. Confirm each game receives one final result.
4. Confirm each completed game has a winner and loser.
5. Confirm team records update.
6. Confirm player stats update.
7. Confirm team stats update.
8. Confirm standings update.
9. Confirm box scores render.
10. Confirm users can view their own results.
11. Advance to the next game day or week.

Game validation:

- No game simulates twice unless explicitly allowed.
- No completed game is missing a score.
- No completed game is missing a winning team.
- No impossible final score exists.
- No tied final score exists unless ties are allowed.
- Winning team receives exactly one win.
- Losing team receives exactly one loss.
- Standings match completed game results.
- Player stat totals match box score totals.
- Team stat totals match player stat totals where applicable.
- Injuries, fatigue, rotation, or lineup effects work if the app has them.

Manual spot checks:

1. Pick 3 teams after several weeks.
2. Count their completed games manually.
3. Compare manual totals to:
   - Team page
   - Standings page
   - Database standings
   - League dashboard

Pass criteria:

- Regular season can complete.
- Every game result is valid.
- Standings match game results.
- No server errors appear during simulation.

Team stats:

- Wins
- Losses
- Runs scored
- Runs allowed
- Win percentage
- Streak
- Home record, if tracked
- Away record, if tracked
- Conference or division record, if tracked

Player stats:

- At-bats
- Hits
- Batting average
- Home runs
- RBI
- Runs
- Walks
- Strikeouts
- Innings pitched
- Earned runs
- ERA
- Pitching strikeouts
- Pitching walks
- Wins and losses for pitchers, if tracked

Invalid stat checks:

- No negative stats.
- No impossible stat totals.
- No duplicate stat rows for the same player and game.
- No player has stats assigned to the wrong team.
- Season totals equal the sum of game logs.
- Box score totals match game totals.
- UI stats match database stats.

Critical rule:

Game logs should be the source of truth. Season totals should equal aggregated game logs.

## 11. Phase 6: Standings and Tiebreakers

- Two teams with the same record.
- Three or more teams with the same record.
- Head-to-head tiebreaker, if supported.
- Run differential tiebreaker, if supported.
- Conference or division tiebreaker, if supported.
- Final standings lock after the regular season.

Validate:

- Standings order is correct.
- Playoff seeding uses the same ordering as standings.
- No team is skipped.
- No team appears twice.
- UI standings and playoff bracket agree.

Pass criteria:

- Tied teams are ordered according to app rules.
- Playoff seeding is explainable and consistent.

## 12. Phase 7: College World Series / Playoff Bracket

At the end of the regular season:

1. Advance to postseason.
2. Generate the College World Series or playoff bracket.
3. Verify the correct teams qualified.
4. Verify seed order.
5. Verify the bracket page renders for all users.
6. Verify eliminated or non-qualified teams have correct access.
7. Simulate the first round.
8. Verify winners advance.
9. Verify losers are eliminated.
10. Simulate each remaining round.
11. Verify the championship matchup is produced correctly.

Bracket validation:

- Each qualifying team appears once.
- No non-qualifying team appears unless rules allow it.
- Every matchup has two teams.
- Winners move to the correct next slot.
- Losers stop advancing.
- Bracket does not regenerate incorrectly after refresh.
- Bracket does not change after games have been played unless intentionally reset by an admin.
- All users see the same bracket state.

Pass criteria:

- Bracket generates correctly.
- Every round advances correctly.
- Championship matchup is valid.

Pass criteria:

- Champion matches the final game winner.
- Final bracket is locked.

- Recruiting data is not corrupted by season completion.

## 14. Phase 9: Abuse, Permissions, and Race Condition Tests

Run these intentionally.

Recruiting:

- Same user opens two tabs and submits recruiting changes at the same time.
- User submits recruiting while admin advances the week.
- User submits negative recruiting points.
- User submits more points than allowed.
- User targets an invalid recruit ID.
- User attempts to recruit for another team.

Games:

- Two sessions try to simulate the same game.
- User refreshes while game simulation is running.
- User attempts to edit a completed game.
- User attempts to simulate a game not involving their team if permissions forbid it.

Postseason:

- User attempts to advance postseason before regular season ends.
- User attempts to regenerate the bracket after games have been played.
- User attempts to simulate an eliminated team's game.
- User attempts to create duplicate championship results.

Expected result:

- Invalid actions are rejected.
- Duplicate actions are ignored or safely blocked.
- No partial writes occur.
- No corrupted standings occur.
- No corrupted recruiting records occur.
- Clear error messages or safe no-op behavior occurs.
- Server logs contain no unhandled exceptions.
